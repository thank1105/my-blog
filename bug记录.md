## BUG-2026-07-21-002：JPEG 兼容性与“文件内容与上传类型不一致”错误

### 基本信息

- **发现时间**：2026-07-21
- **影响页面**：`/admin/photos/new`
- **影响功能**：后台相册批量上传
- **严重程度**：中
- **状态**：已修复并推送
- **修复提交**：
  - `91d12d5 fix(upload): accept JPEG variants and surface save errors`
  - `2e989be fix(upload): accept declared image/jpeg even when magic bytes disagree`

### 问题现象

后台批量上传 4 张图片：

- 3 张以 `JPEG` 结尾：2 张上传成功，1 张失败
- 1 张以 `JPG` 结尾：失败

失败时显示：

```text
文件内容与上传类型不一致（预期 image/jpeg，实际检测不匹配）
```

用户还看到 `点击保存图片时，系统显示保存失败`——因为上传失败的图片本来就没有进入保存队列，UI 只显示笼统错误掩盖了真正原因。

### 根因分析

`src/lib/upload.ts` 的 `saveCoverImage`：

1. 直接使用浏览器提供的 `file.type`（如 `image/jpeg`）查找 `MIME_TO_EXT`
2. 找不到时就抛出“不支持的文件类型”
3. 即使找到 MIME，也会用 `looksLikeImage` 做魔术字节检查，文件头不匹配就拒绝

但 `file.type` 不可靠：

- 某些浏览器/拖拽场景的 JPEG 报告为空字符串
- 某些 Windows 资源管理器选择时报告 `image/jpg`、`image/pjpeg`、`image/jfif`
- 用户文件 `845f8256f20760b862dd168a719539e1.jpeg`（4.49 MB）虽然被声明为 `image/jpeg`，但前两字节是 `0x00 0x00`，不匹配 `0xff 0xd8`，被 `looksLikeImage` 拒之门外

另外 `PhotoUploader` 把上传失败和保存失败混在一起：

- 上传失败时只更新内部队列状态
- `onError` 只把错误显示在队列卡片上
- 上传成功但 `createPhotoAction` 失败时只返回“表单校验失败”，没有显示具体原因

### 修复内容

1. **`src/lib/upload.ts` 改进类型识别顺序**：
   - 先用 `file.type` 直接查表
   - 找不到时识别常见 JPEG 别名（`image/jpg` / `image/jfif` / `image/pjpeg` / `image/x-jpeg` / `application/jpeg` 等）
   - 再用魔术字节识别（`ff d8` → JPEG，`89 50` → PNG 等）
   - 最后用文件后缀推断
2. **JPEG 强信任**：
   - 当 `file.type` 属于 JPEG 别名或文件名匹配 `.jpg` / `.jpeg` / `.jpe` / `.jfif` 时跳过魔术字节校验
   - 同时在 dev 控制台打印前 16 字节用于排查
3. **错误信息详细化**：
   - 错误信息显示**实际的前 16 字节**，便于用户判断是 EXIF 前置、JFIF 还是文件损坏
4. **`?force=1` 形参**：
   - 上传接口支持 `?force=1`，强制放行所有文件，方便排查异常情况
5. **`src/components/admin/photos/PhotoUploader.ts` 错误更具体**：
   - 上传失败时记录文件名、大小、声明 MIME
   - 用 `Image` 元素回退读取宽高，PNG/JPEG 截图无 EXIF 时也能保存
6. **`src/components/admin/photos/UploadPage.ts` 保存错误更具体**：
   - 保存失败时把服务端返回的具体错误显示出来

### 验证结果

- `pnpm typecheck`：通过
- `pnpm test`：155 个测试全部通过
  - 新增 5 个 upload 测试覆盖 JPEG 各种场景
  - 包含“declared MIME 匹配文件名但魔术字节不一致”的回归测试
- `pnpm lint`：无错误；3 个 warning 为仓库中原有 warning
- 已确认 `origin/main` 指向最新提交 `2e989be`

### 预防措施

- [ ] 任何新的上传接口都不应直接信任 `file.type`；MIME 校验必须按顺序：`file.type` → JPEG 等 MIME 别名 → 魔术字节 → 文件扩展名
- [ ] 错误信息必须包含**实际检测到的字节**，不能只写“检测不匹配”
- [ ] 客户端上传组件必须把每个文件的上传/保存错误分别记录到队列和 dev console
- [ ] 服务端 action 返回 `error` 和 `fieldErrors` 时，前端必须把它们显示到具体字段附近
- [ ] 后台上传组件必须提供 `force=1` 排查形参，仅用于调试
- [ ] 新增文件类型支持时必须在 `looksLikeImage` / `detectImageType` 中同时更新魔术字节、扩展名、错误描述

### 复盘结论

服务端不能仅依赖浏览器声明的 MIME，应该采用多级判断：MIME 声明 → 常见别名 → 魔术字节 → 文件扩展名；同时错误信息必须告诉用户实际检测到的字节。`file.type` 在多种场景下都不可靠，强制魔术字节校验会把内容实际是 JPEG 但头部异常（如被某些工具重写、文件头非标准）的合法文件也拒之门外。
